# Backend Learning Playbook

This is a practical checklist for adding backend/database skills to this portfolio without trying to learn every tool at once.

## Current Project Shape

- Frontend: React, TypeScript, Vite.
- Hosting shape: Vercel with serverless routes in `api/`.
- Current database feature: Snake leaderboard.
- Current database provider expected by code: Postgres via `DATABASE_URL`.
- Current contact mail provider expected by code: Resend via `RESEND_API_KEY`.

## Tool Status

- [x] .NET 10 SDK is installed.
- [x] Docker Desktop is installed and running.
- [x] `psql` is installed.
- [x] Azure CLI is installed.

## Recommended Order

1. Install Docker Desktop.
2. Learn Docker by running Postgres locally in a container.
3. Add a `docker-compose.yml` for local Postgres.
4. Connect the existing leaderboard code to local Postgres.
5. Decide whether the production database should stay on Supabase or move to Azure Database for PostgreSQL.
6. Install Azure CLI and learn the Azure portal/CLI basics.
7. Add a .NET backend only if there is a real reason to move away from Vercel serverless routes.
8. Polish small project tasks like the CV PDF sidebar behavior and mail setup.

## Docker

Docker lets the project run services locally without installing every database/server directly on macOS.

Useful first commands after installing Docker:

```bash
docker --version
docker compose version
docker run hello-world
```

## Postgres

For learning, use Postgres in Docker first. It is easy to start, stop, delete, and recreate.

Example local database:

```yaml
services:
  postgres:
    image: postgres:16
    ports:
      - "5433:5432"
    environment:
      POSTGRES_DB: portfolio
      POSTGRES_USER: portfolio
      POSTGRES_PASSWORD: portfolio
    volumes:
      - postgres-data:/var/lib/postgresql/data

volumes:
  postgres-data:
```

Then connect with:

```bash
psql "postgres://portfolio:portfolio@localhost:5433/portfolio"
```

Create the existing leaderboard table:

```sql
create table if not exists snake_leaderboard (
  id bigint generated by default as identity primary key,
  name text not null check (name ~ '^[A-Z]{3}$'),
  score integer not null check (score > 0),
  created_at timestamptz not null default now()
);

create index if not exists snake_leaderboard_score_idx
  on snake_leaderboard (score desc, created_at asc);
```

## Keeping Postgres Updated

There are two different meanings of "constantly update it":

- Updating the app data: the backend writes rows whenever users submit scores, contact entries, etc.
- Updating the database software: Docker pulls newer Postgres images, or Azure handles maintenance windows.

For this project, data updates should happen through backend API routes. Database software updates should be controlled, not automatic without review.

## App Integration

The leaderboard API now reads and writes directly to Postgres using `DATABASE_URL`.

```bash
DATABASE_URL="postgres://portfolio:portfolio@localhost:5433/portfolio"
```

The frontend still calls `/api/leaderboard`, so React does not know anything about the database connection. That is the right boundary: database credentials stay server-side.

## Azure

Azure is useful here for:

- Azure Database for PostgreSQL.
- Azure App Service or Container Apps if you build a .NET API.
- Azure Static Web Apps if you ever move away from Vercel.

Start with:

```bash
az login
az account show
```

## .NET Backend Decision

Do not add .NET just because the SDK exists. Add it when you want to learn:

- REST APIs with ASP.NET Core.
- Entity Framework Core migrations.
- Authentication.
- A backend that can run as a long-lived service instead of serverless functions.

For this portfolio, a good first .NET backend would be a small API that owns the leaderboard:

- `GET /leaderboard`
- `POST /leaderboard`
- Postgres persistence
- Basic validation
- Tests
